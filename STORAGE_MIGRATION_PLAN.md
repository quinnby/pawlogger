# Storage/Persistence Migration Plan
## Final Vehicle-Era Naming Migration Strategy

**Status:** PLANNING PHASE (Not Yet Executed)  
**Created:** March 7, 2026  
**Purpose:** Plan final storage/persistence-layer migration to remove vehicle-era naming  

---

## Executive Summary

This document provides a **migration-planning assessment** for removing vehicle-era naming from the storage and persistence layers of PawLogger. It identifies all remaining storage-critical vehicle terminology, analyzes migration risks, and recommends a pragmatic path forward.

**Key Finding:** Renaming `VehicleId` and related storage identifiers would require **breaking database schema changes**, **API version management**, and **data migration scripts** affecting all users. The **risk significantly outweighs the benefit**.

**Recommendation:** **Path A** - Keep `VehicleId` and related storage identifiers permanently as internal implementation details. Do not execute storage-layer migration.

## 2. Migration Impact Analysis

### 2.1 Database Schema Changes

| Item | Data Affected | Breaking Change | Migration Required | Rollback Risk |
|------|---------------|-----------------|-------------------|---------------|
| PostgreSQL `vehicleId` column | All user record data | **YES** | ALTER TABLE x 18 | **HIGH** |
| LiteDB VehicleId BSON keys | All user record data | **YES** | Data rewrite | **HIGH** |
| Table indexes on vehicleId | Query performance | **YES** | Index rebuild | **MEDIUM** |

**Risk Assessment:**
- **HIGH RISK**: Production databases with millions of records would require downtime or zero-downtime migration strategy
- **Rollback Difficulty**: Once column renamed, old app versions would fail immediately
- **Testing Burden**: Must test migration on PostgreSQL AND LiteDB with various dataset sizes

---

### 2.2 API Contract Changes

| Item | External Impact | Versioning Required | Integration Breakage |
|------|-----------------|---------------------|----------------------|
| Input models (POST/PUT) | **HIGH** | API v2 required | **BREAKS** all clients |
| Export models (GET) | **HIGH** | API v2 required | **BREAKS** all clients |
| Query parameters | **MEDIUM** | Alias possible | **BREAKS** bookmarks |
| WebHook payloads | **HIGH** | Backward compat needed | **BREAKS** integrations |

**Risk Assessment:**
- **BREAKING**: Would require API versioning (v1 vs v2)
- **Maintenance Burden**: Must support both VehicleId and PetId indefinitely for compatibility
- **Documentation**: All API docs would need versioning

## 4. Safe Migration Strategy (If Executing)

**WARNING:** This section describes a safe migration strategy **if** the project chooses to proceed. Based on analysis, **this is NOT recommended** (see Section 5).

---

### 4.1 Compatibility Period Approach

**Phase 1: Alias Layer (6-12 months)**
1. Add new `PetId` property to all models alongside `VehicleId`
2. Dual-write both properties during transition period
3. Database: Add `petId INT` column (nullable initially)
4. API: Accept both `vehicleId` and `petId` in requests
5. API: Return both properties in responses

**Phase 2: Data Migration**
1. Background migration job: Copy `vehicleId` → `petId` for all records
2. Once complete, make `petId` NOT NULL
3. Create indexes on `petId`
4. Monitor API usage to confirm clients are transitioning

**Phase 3: Deprecation (after 12+ months)**
1. Mark `VehicleId` properties as `[Obsolete]`
2. Log warnings when old API endpoints accessed
3. Communicate deprecation timeline to users

**Phase 4: Removal (18-24 months)**
1. Remove `VehicleId` columns/properties
2. Drop old indexes
3. Version bump: v2.0.0 (breaking change)

---

### 4.2 Database Migration Scripts

**PostgreSQL Example:**
```sql
-- Step 1: Add new column
ALTER TABLE app.healthrecords ADD COLUMN petId INT;

-- Step 2: Copy data
UPDATE app.healthrecords SET petId = vehicleId;

-- Step 3: Make NOT NULL
ALTER TABLE app.healthrecords ALTER COLUMN petId SET NOT NULL;

-- Step 4: Create index
CREATE INDEX idx_healthrecords_petId ON app.healthrecords(petId);

-- Step 5 (later): Drop old column
-- ALTER TABLE app.healthrecords DROP COLUMN vehicleId;
```

**LiteDB Migration:**
```csharp
// Read all records
var records = table.FindAll();

// Add PetId property
foreach (var record in records) {
    record.PetId = record.VehicleId;  // Copy value
    table.Upsert(record);
}

// Later: Remove VehicleId property from model class
```

## 5. Migration Decision Paths

### Path A: Keep VehicleId Permanently (RECOMMENDED)

**Strategy:**
- **Do NOT migrate storage/persistence identifiers**
- Accept `VehicleId` as permanent internal implementation detail
- Document this decision for future developers

**Rationale:**
1. **Risk vs. Reward:** Migration benefits are purely cosmetic; risks are substantial
2. **Compatibility:** Avoids breaking external APIs, webhooks, CSV imports
3. **Maintenance Burden:** Eliminates need for dual-property support and versioning
4. **Security:** Avoids authorization bugs from refactoring critical permission logic
5. **Precedent:** Many successful domain migrations keep legacy storage identifiers (WordPress `post_id`, GitHub `repository_id`, etc.)

**What This Means:**
- `VehicleId` remains in database schemas (PostgreSQL column, LiteDB key)
- `GenericRecord.VehicleId` property remains unchanged
- API contracts continue using `vehicleId` property
- Internal code can use semantic names (`petId = record.VehicleId;`)
- User-facing terminology already migrated in earlier phases (UI, labels, docs)

**Implementation:**
- ✅ Add XML documentation comments explaining the naming:
  ```csharp
  /// <summary>
  /// Primary key linking this record to a pet profile.
  /// Note: Property named VehicleId for backward compatibility with existing storage schemas.
  /// </summary>
  public int VehicleId { get; set; }
  ```
- ✅ Update developer documentation to explain legacy naming convention
- ✅ Add inline comments where confusion might occur

**Cost:** Near-zero (documentation only)

**Risk:** Near-zero (no code changes)

---

### Path B: Execute Full Storage Migration

**Strategy:**
- Rename `VehicleId` → `PetId` across database, models, APIs
- Dual-write compatibility period (12+ months)
- API versioning for external contracts
- Data migration scripts for existing installations

**Rationale:**
- "Completeness" of domain migration
- Avoid confusion for future developers seeing `VehicleId` in a pet app

**What This Means:**
- 18+ database tables require ALTER TABLE migrations
- 18+ model classes require property renaming
- 14+ input models require API contract changes
- 10+ export models require API versioning
- 50+ controller methods require parameter renaming
- Authorization model (`UserVehicle`) requires restructuring
- WebHook consumers require updates
- CSV import/export requires format versioning

**Implementation Costs:**
- Engineering: 3-6 months development + testing
- Migration scripts: PostgreSQL + LiteDB dual implementation
- API versioning: Maintain v1 + v2 indefinitely for compatibility
- Documentation: Comprehensive migration guide for users
- Support burden: Field user issues during transition
- Testing: Extensive multi-environment validation

**Risks:**
- **Data Loss:** Migration script bugs could corrupt user data
- **Security Bugs:** Authorization refactoring could expose data to wrong users
- **API Breakage:** External integrations fail without warning
- **Downtime:** Large databases require maintenance windows
- **Rollback Complexity:** Cannot easily revert once columns dropped

**Risk Level:** 🔴 HIGH

## 6. Final Recommendation

### ✅ RECOMMENDED: Path A (Keep VehicleId Permanently)

**Justification:**

1. **User-Facing Migration Complete:** Earlier phases already migrated all user-visible terminology (UI, labels, documentation). End users see "Pet", "Pet Profile", "Pet ID" throughout the application.

2. **Internal Implementation Details:** Storage identifiers like `VehicleId` are internal plumbing. They do not affect user experience or domain understanding.

3. **Industry Precedent:** Many successful applications retain legacy storage identifiers after domain migrations:
   - WordPress: `post_id` (originally for blog posts, now used for pages, custom content)
   - GitHub: `repository_id` (predates "repo" terminology shift)
   - Shopify: `product_id` (used for digital goods, services, etc.)

4. **Cost-Benefit Analysis:**
   - **Benefit of Migration:** Purely aesthetic code consistency (internal only)
   - **Cost of Migration:** 3-6 months engineering + high data/security risk + permanent API versioning burden
   - **Conclusion:** Cost massively outweighs benefit

5. **Non-Breaking Future Path:** If requirements change, Path B can be executed later. Choosing Path A now does not prevent future migration (though it remains inadvisable).
